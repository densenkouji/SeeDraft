# インターフェース改善実装レポート

## 実装日
2026-05-10

## 概要
Voice-to-Textアプリケーションのインターフェースを改善し、文字起こしを原則処理、整形・翻訳を2次処理として明確に分離しました。

## 主な変更点

### 1. バックエンド API の拡張

#### 新しいエンドポイント
- `POST /api/process` - パイプライン処理エンドポイント

#### 新しいデータ構造
```rust
enum PipelineStep {
    Refine { config: RefinementRequest },
    Translate { config: TranslationRequest },
}

struct ProcessPipelineRequest {
    text: String,
    pipeline: Vec<PipelineStep>,
}

struct ProcessPipelineResponse {
    results: Vec<PipelineStepResult>,
}
```

#### 処理フロー
1. クライアントが複数の処理ステップをパイプラインとして送信
2. サーバーが順次処理を実行
3. 各ステップの結果を返却

### 2. UI/UX の改善

#### タブ構成の変更
**変更前（5タブ）:**
- 入力 → 文字起こし → 翻訳 → 整形 → 出力

**変更後（4タブ）:**
- 入力 → 文字起こし → 後処理 → 出力

#### 処理フロー可視化
画面上部に処理フローを表示:
```
音声入力 → 文字起こし → 後処理 → 出力
  ✓必須      ✓必須      ⚙設定可
```

### 3. 後処理タブの統合設計

#### 処理モード設定
- **なし** - 文字起こしのみ
- **整形のみ** - 文字起こし → 整形
- **翻訳のみ** - 文字起こし → 翻訳
- **整形 → 翻訳** - 文字起こし → 整形 → 翻訳

#### 整形設定
- **自動整形**: 文字起こし後に自動実行
- **整形の強度**:
  - 最小限 - 明らかな誤りのみ修正
  - 標準 - 読みやすく整形（デフォルト）
  - 徹底 - フィラー完全除去＋丁寧な文章化
- フィラー除去オプション
- 音声コマンド反映オプション
- スタイル選択（自然文/丁寧/ビジネス/議事録）
- 専門用語・固有名詞の保持
- カスタム指示

#### 翻訳設定
- **自動翻訳**: 文字起こし後に自動実行
- **翻訳タイミング**:
  - 文字起こし直後
  - 整形後（整形ONの場合）
  - 手動のみ
- 翻訳元/先言語選択
- カスタム指示

#### プリセット機能
- **保存**: 現在の設定をlocalStorageに保存
- **読込**: 保存した設定を復元

### 4. JavaScript の改善

#### 新しい関数
- `getPostProcessingConfig()` - 現在の後処理設定を取得
- `buildPipeline()` - 処理モードに基づいてパイプラインを構築
- `executePipeline()` - パイプラインAPIを呼び出して実行
- `executeAutoPostProcessing()` - 文字起こし後の自動処理
- `savePreset()` - 設定をlocalStorageに保存
- `loadPreset()` - 設定をlocalStorageから読込

#### 処理フローの最適化
文字起こし完了後、設定に応じて自動的に後処理を実行:
1. 整形が有効な場合 → 整形を実行
2. 翻訳が有効で、タイミング条件を満たす場合 → 翻訳を実行
3. すべて1回のAPI呼び出しで実行（効率的）

### 5. CSS の追加

#### 新しいスタイル
- `.processing-flow` - 処理フロー表示エリア
- `.flow-step` - 処理ステップの表示
  - `.flow-required` - 必須ステップ（緑）
  - `.flow-optional` - 任意ステップ（黄）
- `.flow-arrow` - ステップ間の矢印
- `.postprocess-section` - 後処理タブのセクション
- `.section-title` - セクションタイトル
- `.preset-buttons` - プリセットボタングリッド

## 使用例

### 例1: 文字起こしのみ
1. 音声ファイルを選択
2. 「後処理」タブで処理モードを「なし」に設定
3. 「文字起こし」ボタンをクリック
→ 文字起こし結果のみが表示される

### 例2: 文字起こし → 整形
1. 音声ファイルを選択
2. 「後処理」タブで処理モードを「整形のみ」に設定
3. 「自動整形」をON
4. 「文字起こし」ボタンをクリック
→ 文字起こし後、自動的に整形が実行される

### 例3: 文字起こし → 整形 → 翻訳
1. 音声ファイルを選択
2. 「後処理」タブで処理モードを「整形 → 翻訳」に設定
3. 「自動整形」と「自動翻訳」をON
4. 翻訳タイミングを「整形後」に設定
5. 「文字起こし」ボタンをクリック
→ 文字起こし → 整形 → 翻訳が順次実行される

### 例4: 手動で後処理を実行
1. 既に文字起こしが完了している状態
2. 「後処理」タブで処理モードを選択
3. 「後処理を実行」ボタンをクリック
→ 現在のテキストに対して後処理が実行される

## 技術的な利点

### 1. パイプライン処理
- 複数の処理を1回のAPI呼び出しで実行可能
- ネットワークオーバーヘッドの削減
- 処理の一貫性が保証される

### 2. 柔軟な設定
- 処理モードで直感的に選択
- タイミング制御で細かい調整が可能
- プリセット機能で設定の再利用が簡単

### 3. 拡張性
- 新しい処理ステップを追加しやすい設計
- パイプラインの順序を柔軟に変更可能
- 各処理の独立性が高い

## 今後の拡張可能性

### 追加可能な処理ステップ
1. **要約** - 長文を簡潔にまとめる
2. **センシティブ情報除去** - 個人情報や機密情報をマスク
3. **感情分析** - 発言のトーンや感情を分析
4. **キーワード抽出** - 重要なキーワードを自動抽出
5. **フォーマット変換** - Markdown、JSON、XMLなどに変換

### UI改善案
1. プリセットの複数保存（名前付き保存）
2. 処理履歴の可視化（どの処理を実行したか）
3. プレビュー機能（実行前に結果を確認）
4. バッチ処理（複数ファイルを一括処理）

## まとめ

この改善により、以下が実現されました：

1. ✅ **原則と2次処理の明確な分離** - 文字起こしは常に実行、整形・翻訳は任意
2. ✅ **直感的なUI** - 処理フローの可視化とタブ統合
3. ✅ **柔軟な設定** - 処理モード、タイミング制御、プリセット
4. ✅ **効率的な処理** - パイプラインAPIによる一括処理
5. ✅ **拡張性の高い設計** - 新しい処理ステップを追加しやすい

ユーザーは自分のワークフローに合わせて、文字起こしから最終出力までの処理を自由にカスタマイズできるようになりました。
